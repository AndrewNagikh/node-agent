# ТЗ: Dashboard управления distributed-llm кластером

## Контекст

Проект — распределённый инференс LLM поверх 3 разнородных домашних машин
(Mac M3 Pro, Mac M1 Pro, Windows RTX 4070 Ti) + оркестратор на отдельном
Linux-хосте. Сейчас всё управление — руками: SSH/RDP на каждую машину,
`git pull`, пересборка, ручной перезапуск процесса, чтение логов через
терминал. Один вечер отладки = десятки ручных циклов "обновил → пошёл на
другую машину → обновил → перезапустил → вернулся проверить".

**Цель дашборда:** одно окно, из которого видно состояние всего кластера
и можно им управлять, не открывая терминал на каждой машине.

**Пользователь:** один (сам автор), не публичный продукт. Не нужна
аутентификация/мультитенантность — но нужна защита от случайного клика
(см. "Опасные действия" ниже).

**Платформа:** веб-дашборд (одна страница, обращается к HTTP API
оркестратора и напрямую к node_agent на каждой ноде — все порты уже
внутри домашней сети, наружу ничего не торчит).

## Экраны

### 1. Обзор кластера (главный экран)

Карточка на каждый узел (3 ноды + оркестратор), видно сразу без клика:
- Имя узла, хост:порт
- Онлайн/офлайн (цветовой индикатор)
- Роль в последней сессии, если есть (entry/middle/final/idle)
- Железо: GPU, VRAM, score (из данных регистрации)
- Состояние воркеров, если сессия активна: entry/middle/final →
  STARTING / READY / FAILED (цветом)
- Кнопка **"Обновить и перезапустить"** прямо на карточке
- Клик по карточке → экран деталей узла

Источник данных: `GET /nodes` (оркестратор) для списка + hardware/score;
`GET /status` на каждый node_agent для worker_states.

### 2. Детали узла

- Всё с карточки, подробнее
- Живые логи (свой node_agent.log + по каждому воркеру: entry/middle/final)
  — источник `GET /debug/log[?worker=entry|middle|final]`, автообновление
  (poll раз в 2-3 сек или явная кнопка "обновить")
- Кнопка **"Обновить и перезапустить"** с подтверждением (см. ниже)
- Прогресс/лог последнего обновления, если было (см. backend-контракт)

### 3. Детали оркестратора

Тот же паттерн, что и узел — своя кнопка обновления, свои логи. Отдельный
экран, потому что оркестратор на другом хосте и физически недоступен
(SSH и то не всегда под рукой), поэтому его апдейт особенно ценен.

### 4. Сессии

- Список активных сессий (модель, layout по нодам, uptime)
- Создать сессию: выбор модели из реестра, опционально
  `speculative_draft_model_url` + `speculative_draft_k`
- Для сессии: отправить промпт (`POST /session/generate`), увидеть
  ответ + timing (tok/s, prefill/decode ms, `speculative: true/false`)
- Удалить сессию (`POST /session/destroy`)

### 5. Модели (можно не в первой версии)

- Список зарегистрированных моделей + статус (`GET /models`)
- Прогресс установки, если идёт

## Backend-контракт для кнопки "Обновить и перезапустить"

Этого эндпоинта **ещё нет**, его нужно спроектировать и реализовать
отдельно (не в рамках дизайна UI, но UI должен закладывать под него
состояния). Предлагаемый контракт:

```
POST /update
  -> { "job_id": "..." }         // fire-and-forget, сразу отдаёт id
     git pull (оба репо) + ./build.sh agents (или orchestrator) +
     перезапуск процесса — выполняется в фоне

GET /update/status?job_id=...
  -> { "state": "pulling|building|restarting|done|failed",
       "log_tail": "..." }        // для live-прогресса в UI
```

Состояния кнопки в UI: обычная → "Обновляю..." (disabled, крутилка) →
успех (зелёная галка на пару секунд) / ошибка (красным, с текстом
последней строки лога и кнопкой "показать полный лог").

**Важно про риск** (не для дизайна, для понимания сценария): если сборка
после обновления сломается, процесс не поднимется — узел "потеряется" из
дашборда до ручного вмешательства на месте. Для дизайна это значит: экран
должен явно показывать "последний известный статус: офлайн с HH:MM" на
карточке узла, а не просто тихо пропадать — чтобы было видно, что
что-то пошло не так, а не что дашборд завис.

## Опасные действия — обязательное подтверждение

Кнопка "Обновить и перезапустить" **всегда** через диалог подтверждения:
- Для ноды: "Обновить и перезапустить node-b? Активные сессии на этом
  узле прервутся." (если есть активная сессия — предупредить явно, не
  просто общей фразой)
- Для оркестратора: "Обновить и перезапустить оркестратор? Дашборд
  временно потеряет связь с кластером на время рестарта."

Без подтверждения — не должно быть ни одного деструктивного действия
(удаление сессии — тоже с подтверждением).

## Стиль и тон

Инструмент для одного технического пользователя, не маркетинговая
страница — приоритет плотности информации и скорости сканирования глазами
над красотой. Тёмная тема по умолчанию (это будет открыто рядом с
терминалами логов). Статусы — обязательно и цветом, и текстом (не
полагаться только на цвет).
